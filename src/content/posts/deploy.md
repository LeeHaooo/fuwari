# Git Clone 后我到底经历了什么

我以前一直以为，Clone 完一个项目，事情就差不多结束了。后来真的把一个博客项目拉到本地，才发现 `git clone` 只是开门，门后面还有一堆我没见过的东西。

## 🤔 一开始的我
![以为 Clone 就等于完成](/images/second1.png)

刚接触 GitHub 的时候，我对它的理解很简单：

```bash
git clone 仓库地址
```

然后项目就会下载到电脑里。

我当时脑子里的流程大概是这样：

```text
Clone
↓
双击运行
↓
结束
```

有点像下载一个压缩包，解压，打开，就能用。

直到我第一次 Clone 了一个博客项目。

## 😅第一次打开项目

![项目文件夹带来的陌生感](/images/second2.png)

项目下载完成后，我打开文件夹，看到一堆陌生名字：

```text
src
public
node_modules
package.json
astro.config.mjs
pnpm-lock.yaml
```

我盯着它们看了半天，心里只有一个问题：

> 这些都是什么？

其中 `package.json` 出现得特别频繁。几乎每个前端项目里都有它，但那时候我完全不知道它负责什么，只觉得它很像一个“项目里不能乱碰的文件”。

## 🚨 第一个报错

我按照 README 里的说明，在终端输入：

```bash
pnpm dev
```

结果终端回了我一句：

```text
pnpm 不是内部或外部命令
```

这也是我第一次意识到：Clone 下来的项目，并不等于已经可以运行。

代码只是被我拿到了。它要跑起来，还缺环境、工具和依赖。

## 💡 依赖到底是什么

![依赖不是代码本体](/images/second3.png)

后来我才明白，前端项目通常会用到很多第三方工具。比如：

- Astro
- TailwindCSS
- Markdown 插件
- 图标库
- 图片处理工具

这些工具不会直接跟着项目代码一起上传到仓库里。仓库里真正保存的是一份清单，告诉你这个项目需要哪些东西、版本大概是什么。

这份清单就写在：

```text
package.json
```

里。

所以 `package.json` 不是神秘文件，它更像是项目的配料表。你拿到了菜谱，还得先把配料买回来。

## 📦安装依赖

README 继续告诉我：

```bash
pnpm install
```

运行之后，项目里终于出现了：

```text
node_modules
```

这个文件夹很大，也很吓人，但它的意思很简单：项目需要的那些第三方包，已经被安装到本地了。

如果拿 Python 类比，`pnpm install` 有点像：

```bash
pip install -r requirements.txt
```

只不过前端项目通常是从 `package.json` 和锁文件里读取依赖信息。

## 🎉第一次运行成功

![从 clone 到 localhost 的完整链路](/images/second4.png)

依赖装好后，我再次执行：

```bash
pnpm dev
```

这次终端里出现了：

```text
localhost:4321
```

我把它复制到浏览器里，打开：

```text
http://localhost:4321
```

博客页面出现的那一刻，我才真正理解，一个项目从 GitHub 到本地跑起来，中间至少要经过这几步：

```text
git clone
↓
安装依赖
↓
运行项目
```

以前我只看到了第一步。

## 🐙开始看懂项目结构

折腾得多了以后，那些一开始看不懂的文件，也慢慢有了位置。

| 文件 | 作用 |
| --- | --- |
| README.md | 项目说明书 |
| package.json | 依赖和脚本配置 |
| pnpm-lock.yaml | 锁定依赖版本 |
| astro.config.mjs | Astro 配置 |
| src | 网站源代码 |
| public | 静态资源 |

以前看到这些文件，我的反应是“完全看不懂”。现在至少知道它们为什么会在这里。

这种感觉还挺微妙的。不是突然变厉害了，而是原来一整团乱麻，开始能分出几根线。

## 🖼 把博客改成自己的

运行成功以后，我开始改一些能看得见的东西：

- 博客名称
- 副标题
- Avatar 头像
- Banner 图片
- GitHub 链接

这些修改都不算复杂，但很有成就感。因为页面一点点从“作者的博客”，变成了“我的博客”。

这也是我第一次觉得，开源项目不是只能远远看着。只要能跑起来，就可以开始拆、改、试。

## 🔥最难忘的一次报错



后来我把博客部署到 Vercel，结果遇到了：

```text
ImageNotFound
```

更奇怪的是，本地运行明明正常，部署上去却失败。

我查了很久，最后发现问题出在图片路径上：

```markdown
![博客头图](public\images\first1.png)
```

我在 Windows 里习惯用反斜杠，也习惯从 `public` 文件夹开始写路径。但网站访问静态资源时，正确写法应该是：

```markdown
![博客头图](/images/first1.png)
```
![本地路径和网站路径](/images/second5.png)

也就是说：

```text
Windows 路径：public\images\first1.png
网站路径：/images/first1.png
```

这个坑让我印象很深。因为它不是代码逻辑错了，而是我把“电脑里的文件路径”和“网页里的资源路径”混在了一起。

## 🚀博客正式上线

当我看到这个地址能在手机上打开时：

```text
https://passerbyblog.vercel.app
```

那种感觉很不一样。

它不再只是：

```text
localhost
```

而是一个真的可以被别人访问到的网站。

那一刻我才觉得，这个博客算是从我的电脑里走出去了。

## 📈这次经历让我学到了什么

![总结](/images/second6.png)

第一，`git clone` 不是终点。

现在如果让我重新描述这个过程，我会写成：

```text
git clone
↓
阅读 README
↓
安装依赖
↓
运行项目
↓
理解结构
↓
修改配置
↓
部署上线
```

这才比较接近一个真实项目的学习过程。

第二，README 真的要看。

以前我会直接跳过 README，觉得它只是说明文字。后来才发现，很多项目已经把运行方式、依赖安装、部署注意事项写在里面了。不是它没告诉我，是我还不会看。

第三，不要太怕报错。

这次搭博客，我遇到过 `pnpm` 报错、路径报错、`ImageNotFound`、部署失败。刚开始每个报错都像一堵墙，后来发现它们更像路标：哪里不懂，哪里就会先爆出来。

解决一次，项目就清楚一点。

## ✍写在最后

如果现在有人问我：Git Clone 后应该做什么？

我会先说一句：

> 别急着运行，先看 README。

Clone 只是把项目带到了电脑里。真正让你理解一个项目的，是安装依赖、跑起来、改一点东西、再处理那些不请自来的报错。

等你能把别人的项目跑起来，并且知道每一步为什么要这么做，离做出自己的项目就真的近了很多。
